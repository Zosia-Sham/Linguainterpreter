# 5. Utilities and State Management

This section covers the helper scripts that maintain the project's state, parse inputs/outputs, and enforce guardrails.

## `src/helpers.py`
**Purpose**: Manages the versioning and saving of generated artifacts (Code, Metrics, Submissions).
**How it works**:
- **`_record_metrics_version()`**: Every time a script runs successfully, it creates a snapshot in `artifacts/versions/<timestamp>_<tag>/` containing the code, the resulting `metrics.json`, and the `submission.csv`.
- **`_append_ledger_row()`**: Appends a summary row to `artifacts/versions/ledger.csv` containing the timestamp, metric score, and paths to the code.
- **`_update_best_from_candidate()`**: Compares new metrics against the historical best. If the new metric is superior, it overwrites the files in the `artifacts/best/` directory.

## `src/utils.py`
**Purpose**: General parsing and string manipulation utilities.
**How it works**:
- **`_parse_yaml_tasks()`**: A robust YAML parser that extracts tasks from LLM responses. If the LLM wraps the response in markdown block quotes (e.g., ````yaml`), it strips them before passing the string to `yaml.safe_load`.
- **`_parse_check_summary()`**: Uses Regex to find `CHECK_SUMMARY: {...}` blocks in the standard output.
- **`clean_specs()`**: A recursive function that removes empty lists, dictionaries, or null values from JSON specs to save LLM context window space.

## `src/validators.py`
**Purpose**: Extracts JSON structures from unstructured LLM and script outputs.
**How it works**:
- **`parse_metrics_from_stdout()`**: Uses a regular expression (`METRICS_JSON\s*:\s*(\{.*\})`) to find and parse the metrics contract output by the generated scripts.
- **`detect_mixed_stacks()`**: A simple Regex-based heuristic that checks if the generated code imports libraries from mutually exclusive or conflicting ecosystems (e.g., importing `torch` and `xgboost` in the exact same script) to prevent bloated environments.

## `src/verification.py`
**Purpose**: Formal static analysis and logical verification of the generated plan and code.
**How it works**:
- **`FormalVerifier` Class**: Implements a form of Bounded Model Checking (BMC).
- **`verify_plan()`**: Uses an LLM to "lift" textual tasks into abstract pre-conditions and effects, ensuring the generated plan is logically sound (e.g., a model cannot be evaluated before the data is loaded).
- **`verify_code_safety()`**: Runs static Regex checks against the generated Python code to detect forbidden patterns, such as `shutil.rmtree('/')` or infinite `while True:` loops. It also uses an LLM to audit the code for destructive operations.

## `src/state.py`
**Purpose**: A simple, persistent key-value store.
**How it works**:
- Reads and writes to a `state.json` file inside the artifacts directory using basic `json.loads` and `json.dumps`. Used for tracking resumes across CLI restarts.

## `src/parsers.py`
**Purpose**: Sanitizes LLM text outputs.
**How it works**:
- **`_strip_think_blocks()`**: Removes `<think>...</think>` tags commonly generated by reasoning models (like DeepSeek or Gemini Pro in specific modes) to isolate the final answer.
- **`extract_code()`**: Isolates code from markdown blocks (` ```python ... ``` `).
- **`extract_boolean()`**: Safely converts conversational text ("Yes, this is correct") to a strict Python boolean.

## `src/config.py`
**Purpose**: Loads and validates the `config.yaml` file.
**How it works**:
- Defines strict Python Dataclasses (`RuntimeConfig`, `LLMConfig`, `OrchestrationConfig`, etc.) to map the YAML structure.
- **`_expand_env()`**: Recursively expands environment variables formatted as `${VAR}` within the YAML values, allowing users to inject API keys dynamically.
