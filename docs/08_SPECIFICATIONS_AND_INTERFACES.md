# 8. Specifications and Interface Definitions

This document details the exact JSON schemas and string contracts used for communication between agents and for persisting the system state.

## 1. Problem Specification (`spec.json`)
This file is generated by `problem_spec_from_text` and represents the "Ground Truth" for the entire project.

**Format Example**:
```json
{
  "modalities": ["image", "tabular"],
  "primary_metric": {
    "name": "f2_micro",
    "maximize": true
  },
  "secondary_metrics": ["precision", "recall"],
  "submission": {
    "columns": ["id", "label"],
    "delimiter": ","
  },
  "validation": {
    "strategy": "stratified_kfold",
    "n_splits": 5,
    "seed": 42
  },
  "constraints": {
    "internet_allowed": false,
    "pretrained_allowed": true,
    "external_data_allowed": false,
    "external_data_requires_tag": false,
    "notes": "Verbatim summary of competition rules (no internet, train from scratch, …)"
  },
  "hardware": {
    "plan": {
      "cpu": { "workers": 4, "threads": 8 },
      "gpu": { "use_cuda": true, "amp": true },
      "batch_size_hint": { "image_bs": 32 }
    }
  },
  "data": {
    "resolved_root": "C:/data/project",
    "meta": { ... }
  }
}
```

## 2. Data Metadata (`data_meta.json`)
Created by `build_data_meta` in `src/data_meta.py`. It provides the "Eyes" for the code-generation LLM.

**Key Fields**:
- `path_aliases`: Mapping of semantic names (e.g., `train_csv`) to absolute filesystem paths.
- `csv_summaries`: Details for each CSV found (delimiter, column names, sampled dtypes).
- `modalities_detected`: List of detected types based on file extensions.
- `notes_tree`: A text-based representation of the directory structure.

## 3. The Task Tree (`tasks_tree.json`)
This is the state file read by the `server.py` dashboard to render the D3.js tree.

**Structure**:
```json
{
  "nodes": {
    "node_id_1": {
      "id": "node_id_1",
      "parent_id": null,
      "label": "Main Task",
      "status": "done",
      "task_text": "...",
      "children": ["child_id_2"]
    },
    "child_id_2": {
      "id": "child_id_2",
      "parent_id": "node_id_1",
      "label": "Load Data",
      "status": "running",
      "task_text": "..."
    }
  },
  "root_id": "node_id_1"
}
```

## 4. Stdout Signaling Contract (`METRICS_JSON`) and persistence

Generated Python scripts **must** print a specific JSON line to stdout. The **same object** (same keys) should be written to **`artifacts/metrics.json`** when `type` is `calculated` (see `PERFORM_TASK_PYTHON_SYS` in `src/prompts/templates.py`).

**Canonical keys (calculated)** — aligned with `src/utils.py` **`_validate_and_normalize_metrics`** (orchestrator adds/normalizes `type` if missing):
- `type`: `"calculated"`
- `primary`: float — the scalar being optimized (not `primary_score` or `score`)
- `name`: string — must match `spec.primary_metric.name`
- `maximize`: boolean — must match `spec.primary_metric.maximize`
- `extras`: optional object — diagnostics (floats, `*_path` strings to files under `artifacts/`)

**Skipped (EDA / load-only)**:
- `type`: `"skipped"`
- `reason`: e.g. `"task_does_not_produce_metrics"`

**Example (calculated)**:
```text
METRICS_JSON: {"type": "calculated", "primary": 0.85, "name": "accuracy", "maximize": true, "extras": {"f1_macro": 0.82}}
```

**Parsing & recovery** (`src/validators.py`):
- **`parse_metrics_from_stdout`**: finds `METRICS_JSON:` and extracts a **brace-balanced** JSON object; if multiple prints exist, the **last valid** `calculated`/`skipped` wins (training logs often prepend noise).
- **`metrics_recover_from_stdout`** (`src/prompts_agents.py`): optional **LLM** pass on a **tail** of stdout when regex parsing fails but logs look like training/eval — then **`validate_recovered_metrics`**; if still missing, **`verification_code_gen`** runs.

**Ledger** (`src/helpers.py`):
- Valid **calculated** metrics trigger **`_update_best_from_candidate`** → `artifacts/last/`, `artifacts/versions/<ts>_<tag>/metrics.json`, **`artifacts/versions/ledger.csv`** (and `index.json`). **Skipped** runs do not append numeric rows to the ledger.

## 5. Bootstrapping and Environments
The system manages the local execution environment through `src/bootstrap.py` and the `GlobalOrchestrator`.

- **`bootstrap_gpu_stack`**: 
    1. Checks `config.yaml` for `preinstall.enable`.
    2. Installs `torch` and `torchvision` using the specified `torch_cuda_index_url` (crucial for GPU support on Windows).
    3. Installs extra packages defined in `preinstall.pkgs`.
    4. Runs a `probe_cuda.py` script to verify that the installed PyTorch can actually see the GPU.
- **`pip_install`**: A method in the orchestrator that calls `pip install` inside the project's virtual environment. It handles `index-url` and `--extra-index-url` parameters to ensure libraries like `xgboost` or `catboost` are installed correctly for the detected hardware.

## 6. Logging and Traceability
- **CLI run logs**: `main.py` tees stdout/stderr to `{project_root}/{logs_dir}/run_YYYYMMDD_HHMMSS.log` and `last_run.log` (unless `--no_log_file`).
- **Per-step execution**: Shell/Python runs may also be logged under `artifacts/logs/` depending on orchestrator settings.
- **Error Traces**: When `ErrorRouter` triggers, it saves the faulty code and the full stderr trace to allow the "Lead Agent" to diagnose the issue.
- **Versions Index**: `artifacts/versions/index.json` maintains a historical record of every single script that produced a valid `METRICS_JSON`, allowing the system to point back to any previous state during a `--resume`.
